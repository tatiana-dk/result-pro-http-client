**Минимальный API твоей библиотеки выглядит примерно так (простыми словами):**

### Как создаётся клиент с базовыми настройками
Создаёшь один раз объект-клиент и говоришь ему:
- базовый адрес сайта (baseURL)
- дефолтные заголовки (например, всегда отправлять Content-Type: application/json)
- дефолтный таймаут

### Как отправляешь запросы
Вызываешь простые методы на клиенте:
- client.get('/users')
- client.post('/login', { username, password })
- client.put('/profile', updatedData)
- client.delete('/post/42')

Или более гибко:
client.request({ method: 'PATCH', url: '/settings', body: newSettings })

### Как настраиваешь заголовки
Три уровня настройки (от самого общего к конкретному):
1. При создании клиента — для всех запросов
2. При конкретном запросе — перекрывает дефолтные
3. Для одного запроса можно передать headers прямо в объект

### Как задаёшь таймаут
Два способа:
- глобально при создании клиента (например, 10 секунд)
- для конкретного запроса (перекрывает глобальный)

### Как отменяешь запрос
Передаёшь в запрос специальный сигнал отмены (abort signal).  
Если захотел отменить — вызываешь .abort() на этом сигнале → запрос прерывается.

### Простыми словами важные понятия

- **Таймаут** — это как таймер на микроволновке. Если сервер слишком долго думает и не отвечает за 8 секунд — библиотека сама прерывает запрос и говорит «слишком долго ждали».
- **Отмена запроса** — это кнопка «стоп» прямо во время загрузки. Например, пользователь нажал «назад» на странице поиска — можно мгновенно отменить запрос, чтобы не тратить трафик и не ждать ответа зря.
- **Повтор (retry)** — если запрос упал по «плохой» причине (сервер сказал 503, 429, сеть пропала на секунду) — библиотека может автоматически попробовать ещё раз (1–3 попытки) вместо того чтобы сразу показать ошибку пользователю.

### 5 типичных сценариев использования (без кода)

1. Приложение показывает список постов — при загрузке страницы делается один get-запрос с базовым адресом и токеном в заголовках.
2. Пользователь редактирует профиль и нажимает «Сохранить» — отправляется put-запрос с таймаутом 7 секунд.
3. При поиске товаров в магазине пользователь быстро меняет фильтры 5 раз подряд — предыдущие запросы автоматически отменяются.
4. Мобильное приложение на слабом интернете пытается загрузить аватарку — если не получилось за 5 секунд, запрос повторяется ещё раз автоматически.
5. Форма логина — если сервер ответил 401 (неверный пароль), повтор не делается, а если 503 (сервер временно недоступен) — делается 2 автоматических повтора с небольшой паузой.
